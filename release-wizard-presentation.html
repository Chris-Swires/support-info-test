The Problem: Releases Are Too Hard

Expert-Only Process

Running a Managed Service extension release requires deep knowledge of SRS, Snowstorm, Jira, and RVF. NRC Release Leads cannot act independently without SI involvement at every step.

Inconsistent Execution

The release process is documented in procedures and tribal knowledge. Steps are performed manually, in different orders, with varying checks — leading to inconsistencies and missed pre-conditions.

Fragmented Tooling

Jira tickets, Snowstorm branches, SRS builds, and RVF reports all live in separate systems. Coordination requires manually context-switching and copy-pasting between at least four different tools.

The Goal

Replace ad-hoc, expert-gated release coordination with a guided, step-locked wizard that enforces the correct sequence, automates checks, creates and updates Jira tickets, monitors builds in real time, and provides plain-language guidance — so that a non-expert NRC Release Lead can successfully execute a release end-to-end.

The Release Wizard: Three Phases

1
Phase 1

Preparation

NRC → SI Intake. Raise the Jira release ticket, verify country and dates, select the release type, validate the branch, run the automated pre-release audit, and assign the NRC Release Lead.

  • Release Request raised in Jira
  • SI Intake: country, dates, summary
  • Standard or Frequent release path
  • Automated 4-point pre-release audit
  • NRC Release Lead assigned
2
Phase 2

Process

Checks & Builds. Infrastructure health verification, RTZ decision point, pre-release checklist sign-off, then initiation of the SRS build pipeline — monitored live via WebSocket.

  • SRS + Snowstorm health checks
  • RTZ update decision point
  • Pre-release checklist confirmation
  • Standard or Frequent build triggered
  • Real-time build progress via WebSocket
3
Phase 3

Handover

Cleanup & BAU. RVF validation results, SCTID versioning, production build, final publication, NRC notification, master project confirmation, and clean return-to-BAU state.

  • RVF results surfaced in plain language
  • SCTID versioning monitored live
  • Production build triggered and monitored
  • Build published, Jira ticket → Released
  • NRC notified, session cleared to BAU

Phase 1: Preparation in Detail

1

NRC Release Request

One button creates a structured Jira ticket with the correct fields, labels, and format. The ticket key is captured immediately for all subsequent updates.

2

SI Intake — Country & Summary

ISO 3166-1 validated country code, release month and year. Auto-formats the Jira ticket summary as CC | Month | Year | Release and saves to state.

3

Release Type Selection

Guided card selection between Standard Release (extended verification, multi-day pre-prod) and Frequent Release (accelerated, delta-focused). Routes the entire downstream workflow accordingly.

4

Branch Config & Pre-Release Audit

Branch path auto-suggested from country code, verified against Snowstorm. Dependency date auto-populated from branch metadata. Automated 4-point audit runs with animated checklist feedback.

Phase 2: Process & Build Pipeline

5

Infrastructure Health Check

Three parallel checks on entry to Phase 2: SRS API health, Snowstorm API health, and SRS WebSocket connectivity. All must pass before proceeding. Retry option on failure, with RAD escalation link.

5

RTZ Decision Point

Plain-language yes/no prompt: does this extension require RTZ updates before release? If yes, workflow blocks and directs the user to complete RTZ in the Authoring Platform first.

6

Pre-Release Checklist

Four manual confirmations required before build can start: authoring complete, classification run and saved, validation passes, all CRS requests resolved. Each item has a contextual help tooltip.

7

Build Initiation (Standard or Frequent)

Calls SrsService.startStandardRelease() or startFrequentRelease() based on Phase 1 selection. Stores build ID, transitions Jira ticket, navigates to live build monitor.

System Integrations

Jira (REST API v3)

Full lifecycle management of the MS release ticket. The wizard creates, transitions, comments on, and assigns the Jira ticket at every workflow step — providing a complete audit trail.

createIssue transitionIssue addComment assignIssue searchIssues (JQL)

Snowstorm (Terminology Server)

Branch verification, metadata discovery, refset population checks, and code system lookup. All Snowstorm calls are read-only — safe to run in dry-run mode without side effects.

branchExists getBranchMetadata checkRefsetPopulation listCodeSystems

SRS (SNOMED Release Service)

Build triggering, status polling, and real-time log streaming via STOMP over SockJS WebSocket. Handles Standard and Frequent release paths, versioning, and final publication to stable channel.

triggerBuild getBuildStatusStream publishBuild tagBuildAsPublished

IMS (Identity Management)

Session validation on load, role resolution for RBAC enforcement, and user display details for the header and assignment features. Redirect to IMS login on session expiry.

validateSession getRoles getAccountDetails NRC_RELEASE_LEAD · SI_RELEASE_MANAGER

WebSocket integration (STOMP/SockJS) drives the live build log feed and progress indicators. SRS build status events arrive in real time without polling — auto-advancing the wizard to the next step on COMPLETE.

Safety, Resilience & Non-Expert UX

Role-Based Access

Two roles derived from IMS: NRC_RELEASE_LEAD (Phases 1 & 3) and SI_RELEASE_MANAGER (Phases 2 & 4/5/6). Unauthorised actions show a plain-language explanation. Admin override available.

Step gating enforced per role

Dry Run Mode

Available from the Phase 2 gate. All Jira and SRS writes are replaced with mock responses. Snowstorm calls still execute (read-only). A persistent amber banner reads "DRY RUN ACTIVE". All log entries prefixed with [DRY RUN].

Safe for onboarding & testing

Session Recovery

Workflow state persists to sessionStorage. If the user refreshes or navigates away mid-release, the wizard offers to Resume (restores step) or Start Fresh. No release is ever silently lost.

Survives accidental refresh

Global Error Boundary

All unhandled HTTP errors and WebSocket disconnections are caught, logged in red to the automation log, shown as a notification toast, and surfaced as a plain-language panel with a Retry button and a clearly-labelled "View in RAD (expert tool)" escalation link.

Automation Log + Jira Sync Panel

Every action is timestamped and logged in the sidebar terminal widget (command / info / error severity). A separate Jira sync panel polls the release ticket every 30 seconds and surfaces any external status changes with a toast notification.

Architecture & Technical Stack

Angular (Standalone)

  • Standalone component architecture — no NgModules
  • TypeScript strict mode throughout
  • Typed service layer with Observable<T> contracts
  • Functional route guards (canActivate)
  • esbuild pipeline — fast hot-reload

Tailwind CSS v4

  • SNOMED International brand tokens as CSS custom properties
  • Utility-first — no custom SCSS files
  • Dark/light mode via Tailwind's dark: variant
  • PostCSS integration with esbuild
  • Fira Code monospace for automation log

Real-Time: RxJS + STOMP

  • STOMP over SockJS for SRS WebSocket
  • Subscribe to /topic/build.{buildId}
  • Typed BuildStatusUpdate model
  • Reconnection + cleanup on destroy
  • Jira sync via 30s polling Observable
