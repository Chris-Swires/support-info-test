Background

What is GMDN?

The GMDN Agency

The Global Medical Device Nomenclature (GMDN) Agency is a registered UK charity and not-for-profit organisation, responsible for the creation and maintenance of the globally recognised standard for naming and identifying medical device types. It operates under WHO and EU guidance as the internationally endorsed nomenclature.

History & Purpose

GMDN emerged from the need for a single, internationally agreed nomenclature across regulatory, supply chain, and post-market surveillance domains. It is mandated for Unique Device Identification (UDI) submissions in the EU (MDR/IVDR), USA (FDA), and many other jurisdictions. Adopted by WHO as the preferred medical device naming standard.

Global Adoption

GMDN terms are used by regulators, manufacturers, healthcare providers, and procurement systems across 100+ countries. The database is continuously updated as new device types emerge, with version 26.2 (February 2026) containing over 31,000 active device type definitions.

Regulatory Role

Used extensively in UDI databases (EUDAMED, FDA GUDID), procurement catalogues, adverse event reporting, and post-market surveillance. GMDN provides the "what type of device is this?" answer that complements UDI's device-instance identification — bridging regulatory and clinical worlds.

Structure

How GMDN is Organised

①

Five-Digit Code

Sequential cardinal numbers starting from 10000. Codes are opaque — they carry no inherent hierarchy or category information.

47264
e.g. Cardiac pacemaker, implantable
②

Term Name

A short, generic, internationally agreed descriptor. Constructed using established naming conventions — noun first, followed by qualifying adjectives (e.g. Catheter, urinary, single-use).

③

Definition

Each term includes a definition describing the clinical intended use and clinically relevant attributes of the device type represented. Definitions are plain English and device-class scoped, not device-instance scoped.

Classification & Categories

GMDN codes are not inherently hierarchical — the five-digit number alone conveys no structural information. However, the Agency maintains a multi-level category tree used for browsing and reporting. Category membership is managed as metadata alongside the term. This flat-code-plus-category model is why a custom IS-A hierarchy must be constructed when importing GMDN into a terminology server like Snowstorm.

Partnership

GMDN Agency & SNOMED International

January 2026 Collaboration

SNOMED International and the GMDN Agency announced a strategic collaboration to advance global medical device content and interoperability, strengthening the connection between clinical and regulatory medical device data. The partnership aims to support safer, more efficient, and more interoperable health systems worldwide.

Prior Collaboration History

The January 2026 agreement builds on existing work: relevant GMDN content has already been incorporated into SNOMED CT International Edition, and a formal map from SNOMED CT concepts to GMDN codes has been produced and maintained. New device content continues to be included in each SNOMED CT International release.

Collaboration Goals

Enable unified clinical + regulatory device data across the care lifecycle

Support post-market surveillance, traceability, procurement & inventory management

Bridge long-standing gaps between clinical documentation and regulatory reporting

This extension import project directly supports the collaboration by making the full GMDN v26.2 dataset browsable and queryable within Snowstorm, enabling richer device terminological cross-referencing alongside SNOMED CT International content.

Design Decision

Core Architecture

Root Placement

A single GMDN Root Concept is created as a child of 260787004 |Physical object|. All 31,092 GMDN device terms are then created as direct children of this root, forming a two-level IS-A hierarchy within SNOMED CT.

Physical object (260787004)
└── GMDN Root
    ├── Abdominal binder, single-use
    ├── Cardiac pacemaker, implantable
    └── … (31,092 terms)

ID Generation Strategy

Component IDs (concept, description, OWL axiom, language refset) are left blank in the RF2 files. Snowstorm auto-generates valid SCTIDs from the branch configuration on import. This avoids the need for a pre-allocated namespace and keeps the pipeline simple.

OWL axioms that reference the GMDN Root concept ID face a bootstrapping challenge — the root's ID is unknown at generation time. A two-pass approach or known root ID is required to resolve cross-references.

Flat Hierarchy Rationale

Because GMDN's category structure is metadata-driven rather than code-driven, the import does not attempt to recreate GMDN categories as SNOMED CT parent concepts. A flat hierarchy under a single root preserves the integrity of the original GMDN model while making all terms findable via SNOMED CT ECL (e.g. << GMDN Root). Category-level relationships could be added in a future pass using the GMDN category metadata.

Design Decision

Description Modelling

Each GMDN term generates three descriptions per concept, capturing the SNOMED CT FSN convention, a bare preferred synonym, and the full GMDN definition.

900000000000003001

Appends the SNOMED CT semantic tag (physical object) to the GMDN term name. Uniquely identifies the concept within the hierarchy.

single-use (GMDN)"
900000000000013009

The bare GMDN term name, without semantic tag. This is the human-readable preferred term used in search and display.

The full GMDN definition, preserved verbatim. Describes the clinical intended use and device attributes of the term.

Design Decision

OWL Axioms & Status Handling

OWL Axioms for IS-A

Hierarchy is expressed using OWL axiom expressions (sct2_sRefset_OWLExpression) rather than stated relationships. This aligns with current SNOMED CT best practice and ensures Snowstorm's reasoner can consume the extension correctly.

// Each GMDN term concept gets:
SubClassOf(:<GMDN_term_ID>
  :<GMDN_root_ID>)

Obsolete GMDN Terms

GMDN terms marked obsolete in the source XML are handled carefully to preserve historical data while correctly reflecting their retired status.

Concept row: active = 0 (inactive)
OWL axiom: active = 0 (inactive)
Descriptions: remain active = 1 (searchable)

Keeping descriptions active on inactive concepts preserves the ability to search for retired device names — important for historical record linkage and data provenance.

Definition Status

All concepts use 900000000000074008 (Primitive) — appropriate since GMDN terms carry only a generic IS-A relationship, with no defining attribute modelling from SNOMED CT.

Import Strategy

Option: Import Full Content into the Extension

A pragmatic alternative to reconciling GMDN content against the SNOMED CT International Edition is to ignore the international content entirely and import the full GMDN dataset directly into the extension — including any device terms that already exist in some form within SNOMED CT International.

What About Duplication?

At first glance, importing all GMDN terms regardless of overlap with SNOMED CT International looks like duplication. In practice, there will be conceptual overlap — both datasets contain terms for common device types such as catheters, pacemakers, or infusion sets.

Conceptual overlap ≠ strict SNOMED duplication. The two sets of concepts describe the same real-world devices but are modelled independently.

Why It's Valid in Strict SNOMED Terms

  • FSNs will differ. The GMDN extension uses the semantic tag (GMDN), making every FSN unique and distinct from any SNOMED CT International FSN — no duplicate FSN can exist by construction.
  • No attribute modelling. GMDN concepts are Primitive with IS-A only; there are no defining relationships that would conflict with or replicate the formal modelling of SNOMED CT International device concepts.
  • Extension is Primitive. Because definition status is Primitive throughout, the extension makes no completeness claims — no reasoner will conflate these concepts with International equivalents.

Trade-off Summary

This approach trades conceptual tidiness for implementation simplicity. The extension will contain GMDN concepts that are semantically related to — but formally independent of — SNOMED CT International device concepts. Users querying devices will see both; a future association refset or SNOMED CT → GMDN map could be used to link them explicitly where a precise correspondence is known.

Import Strategy

Option: GMDN Extension + Inactivate International Duplicates

A more curated alternative: import all GMDN terms into the extension as before, but also inactivate the corresponding SNOMED CT International device concepts within the extension. The result is a combined view in which every device type is represented by exactly one active concept — the GMDN-sourced one.

1

Import GMDN Extension

All 31,092 GMDN terms are loaded as active concepts under the GMDN root, exactly as described in the full-content approach.

2

Identify International Duplicates

Map GMDN terms to their corresponding SNOMED CT International device concepts — using the existing SNOMED CT → GMDN map and manual review for unmapped overlaps.

3

Inactivate in Extension

For each matched International concept, add an extension row to the Concept file with active = 0, overriding its active status within this extension's branch.

The Outcome

When browsing the extension branch in Snowstorm, the combined active content contains one concept per device type. The GMDN concept is active; the corresponding International concept is inactive within this branch.

● Cardiac pacemaker (GMDN) — active
● Cardiac pacemaker (Intl.) — inactive in ext.

Significant Considerations

  • Extension scope only. Inactivation applies within the extension branch; the International Edition concepts remain active in their own release. Any system consuming the International Edition unmodified will not see this.
  • Existing data using International concepts. Clinical records coded to the now-inactivated International concept will require migration or association refset guidance to link them to the replacement GMDN concept.
  • Maintenance burden. Every future SNOMED CT International release that adds new device concepts must be reviewed and potentially inactivated against the GMDN set.
  • Inactivation reason. A suitable inactivation indicator and historical association (e.g. Replaced By → GMDN concept) should be provided so downstream consumers understand the intent.

Next Steps

Remaining Work & Path Forward

1

Resolve OWL Cross-Reference

The OWL axioms for child concepts need to reference the GMDN Root concept SCTID. Two approaches: (a) two-pass generation — import the root first, retrieve its assigned ID, then regenerate child axioms; or (b) pre-assign a known root ID from the branch namespace before generating the package.

2

Snowstorm Batch Import & Validation

Load the finalized RF2 package into Snowstorm via the batch RF2 import endpoint. Verify that all 31,092 concepts are active and browsable, that the hierarchy resolves under 260787004 |Physical object|, and that ECL queries return expected results.

3

Category Hierarchy (Optional Enhancement)

Use GMDN category metadata to create intermediate parent concepts, enabling richer browsing (e.g. Cardiovascular Devices → individual device terms). This would move from a flat two-level hierarchy to the full GMDN multi-level category structure within SNOMED CT.

4

Release & Versioning

Establish a process for keeping the extension in sync with GMDN releases (currently on a quarterly cadence). Consider aligning GMDN extension release dates with SNOMED CT International release cycles or maintaining an independent release schedule.

Once live in Snowstorm, the GMDN extension enables SNOMED CT ECL queries that span both clinical terminology and device nomenclature — a key enabler for the GMDN Agency / SNOMED International strategic collaboration goals.
